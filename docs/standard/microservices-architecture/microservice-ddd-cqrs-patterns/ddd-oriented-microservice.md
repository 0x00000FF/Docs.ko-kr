## <a name="keep-the-microservice-context-boundaries-relatively-small"></a><span data-ttu-id="bfdbb-120">마이크로 서비스 컨텍스트 경계를 상대적으로 작게 유지</span><span class="sxs-lookup"><span data-stu-id="bfdbb-120">Keep the microservice context boundaries relatively small</span></span>

<span data-ttu-id="bfdbb-121">경계가 지정된 컨텍스트 사이에서 어디를 경계로 할 것인지 결정하는 것은 경합하는 두 목표의 균형을 맞추는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-121">Determining where to place boundaries between Bounded Contexts balances two competing goals.</span></span> <span data-ttu-id="bfdbb-122">먼저 가능한 가장 작은 마이크로 서비스를 만들려고 합니다. 이것이 주 목표는 아니며 응집이 필요한 항목 주변의 경계를 만들어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-122">First, you want to initially create the smallest possible microservices, although that should not be the main driver; you should create a boundary around things that need cohesion.</span></span> <span data-ttu-id="bfdbb-123">두 번째로 마이크로 서비스 간의 자잘한 통신은 지양하고자 합니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-123">Second, you want to avoid chatty communications between microservices.</span></span> <span data-ttu-id="bfdbb-124">이 두 목표는 서로 상충할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-124">These goals can contradict one another.</span></span> <span data-ttu-id="bfdbb-125">추가적으로 경계가 지정된 새 컨텍스트를 분리할 때마다 매번 통신 경계가 신속하게 증대하는 수준이 되도록 시스템을 가능한 작은 규모의 여러 마이크로 서비스로 나누어 그 균형을 맞추어야 합니다. </span><span class="sxs-lookup"><span data-stu-id="bfdbb-125">You should balance them by decomposing the system into as many small microservices as you can until you see communication boundaries growing quickly with each additional attempt to separate a new Bounded Context.</span></span> <span data-ttu-id="bfdbb-126">응집력은 경계가 지정된 단일 컨텍스트에서 핵심입니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-126">Cohesion is key within a single bounded context.</span></span>

<span data-ttu-id="bfdbb-159">Eric Evans는 훌륭한 저서인 [Domain Driven Design](https://domainlanguage.com/ddd/)에서 도메인 모델 게층과 응용 프로그램 계층을 따르는 것에 대해 설명한 바 있습니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-159">Eric Evans's excellent book [Domain Driven Design](https://domainlanguage.com/ddd/) says the following about the domain model layer and the application layer.</span></span>

<span data-ttu-id="bfdbb-160">**도메인 모델 계층**: 비즈니스 개념, 비즈니스 상황 정보, 비즈니스 규칙을 나타내는 작업을 담당합니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-160">**Domain Model Layer**: Responsible for representing concepts of the business, information about the business situation, and business rules.</span></span> <span data-ttu-id="bfdbb-161">저장을 위한 기술 세부 정보는 인프라에 위임되지만 비즈니스 상황을 반영한 상태가 여기서 제어 및 사용됩니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-161">State that reflects the business situation is controlled and used here, even though the technical details of storing it are delegated to the infrastructure.</span></span> <span data-ttu-id="bfdbb-162">이 계층은 비즈니스 소프트웨어의 핵심입니다.</span><span class="sxs-lookup"><span data-stu-id="bfdbb-162">This layer is the heart of business software.</span></span>
