# 매개 변수 디자인
이 섹션에 인수를 확인 하기 위한 지침과 섹션을 포함 하 여 매개 변수 디자인 광범위 한 지침을 제공 합니다. 에 설명 된 지침을 참조 해야 하는 또한 [매개 변수 이름 지정](../../../docs/standard/design-guidelines/naming-parameters.md)합니다.  
  
 **✓ 수행** 멤버에 필요한 기능을 제공 하는 최소 파생 된 매개 변수 형식을 사용 합니다.  
  
 예를 들어 컬렉션을 열거 하 고 각 항목을 콘솔에 출력 하는 메서드를 디자인 하려고 한다고 가정 합니다. 이러한 메서드를 취해야 <xref:System.Collections.IEnumerable> 매개 변수로 하지 <xref:System.Collections.ArrayList> 또는 <xref:System.Collections.IList>, 예를 들어 있습니다.  
  
 **X 하지 않으려면** 예약 된 매개 변수를 사용 합니다.  
  
 멤버에 대 한 더 많은 입력 이후 버전에서 필요한 경우에 새 오버 로드를 추가할 수 있습니다.  
  
 **X 하지 않으려면** 포인터, 포인터, 배열 또는 다차원 배열을 매개 변수로 받는 메서드를 공개적으로 노출 합니다.  
  
 포인터 및 다차원 배열 어렵습니다 상대적으로 올바르게 사용 하 여입니다. 거의 모든 경우에 이러한 형식을 매개 변수로 입지 않도록 Api 다시 설계 수 있습니다.  
  
 **✓ 수행** 모든 배치 `out` 매개 변수에서 값을 모두 수행 하 고 `ref` 매개 변수 \(매개 변수 배열 제외\) 매개 변수 오버 로드 사이에 불일치가 발생 하는 경우에 \(참조 [멤버 오버 로드](../../../docs/standard/design-guidelines/member-overloading.md)\).  
  
 `out` 메서드 시그니처를 더욱 쉽게 이해 함께 그룹화 하 고 추가 반환 값으로 매개 변수를 볼 수 있습니다.  
  
 **✓ 수행** 멤버를 재정의 하는 경우 매개 변수 이름 또는 인터페이스 멤버 구현 일관 되 게 합니다.  
  
 이 메서드 간의 관계를 더 잘 통신합니다.  
  
### 열거형과 부울 매개 변수 중에서 선택  
 **✓ 수행** 구성원 부울 매개 변수 두 개 이상의 할 경우 열거형을 사용 합니다.  
  
 **X 하지 않으려면** 될 수 없으므로 두 개 이상의 값에 대 한 요구를 확실히 경우가 아니면 부울 값을 사용 합니다.  
  
 열거형 값의 향후 추가 하기 위해 약간의 공간 제공 하지만 모두에 영향을 설명 하는 열거형 값을 추가 알고 있어야 [열거형 디자인](../../../docs/standard/design-guidelines/enum.md)합니다.  
  
 **✓ 고려** 진정으로 두 가지 상태 값이 고 부울 속성을 초기화 하기 위해서만 사용 하는 생성자 매개 변수에 대 한 부울 값을 사용 하 여 합니다.  
  
### 인수 유효성 검사  
 **✓ 수행** public, protected 또는 명시적으로 구현 된 멤버에 전달 된 인수의 유효성을 검사 합니다. Throw <xref:System.ArgumentException?displayProperty=fullName>, 또는 해당 서브 클래스 중 하나를 유효성 검사에 실패 합니다.  
  
 Public 또는 protected 멤버 자체에서 발생 하는 실제 유효성 검사 반드시 없는 참고 합니다. 일부 private 또는 internal 루틴에서 낮은 수준에서 발생할 수 있습니다. 주요 요점은 최종 사용자에 게 노출 되는 전체 화면 영역에서 인수를 검사 하는 것입니다.  
  
 **✓ 수행** throw <xref:System.ArgumentNullException> null 인수가 전달 되 고 멤버 null 인수를 지원 하지 않습니다.  
  
 **✓ 수행** 열거형 매개 변수 유효성을 검사 합니다.  
  
 Enum 인수가 enum으로 정의 된 범위에 있을 것을 가정 하지 마십시오. CLR은 열거형에 값이 정의 되지 않은 경우에 enum 값에 모든 정수 값을 캐스팅할 수 있습니다.  
  
 **X 하지 않으려면** 사용 <xref:System.Enum.IsDefined%2A?displayProperty=fullName> 열거형 범위를 확인 합니다.  
  
 **✓ 수행** 유효성을 검사 한 후 변경할 수 있는 인수 변경 있을 수 있습니다.  
  
 멤버는 중요 한 보안을 모르는 경우 복사본 및 다음의 유효성을 검사 하 고 인수를 처리 하는 것이 좋습니다.  
  
### 매개 변수 전달  
 프레임 워크 디자이너의 관점에서 그룹이 세 가지 주요 매개 변수: 값으로 매개 변수를 `ref` 매개 변수 및 `out` 매개 변수입니다.  
  
 인수에 값으로 매개 변수를 통해 전달 될 때는 멤버에 전달 되는 실제 인수의 복사본을 받습니다. 인수가 값 형식, 인수의 복사본이 스택에 배치 됩니다. 인수가 참조 형식, 참조의 복사본이 스택에 배치 됩니다. 가장 인기 있는 CLR 같은 언어를 값으로 매개 변수를 전달 하려면 C\#, VB.NET 및 c \+ \+에서는 기본입니다.  
  
 인수를 통해 전달 되 면는 `ref` 매개 변수를 멤버에 전달 되는 실제 인수에 대 한 참조를 받습니다. 인수가 값 형식, 인수에 대 한 참조를 스택에 배치 됩니다. 인수가 참조 형식, 참조에 대 한 참조를 스택에 배치 됩니다.`Ref` 호출자에 의해 전달 된 인수를 수정 하는 멤버를 허용 하도록 매개 변수를 사용할 수 있습니다.  
  
 `Out` 매개 변수는 유사한 `ref` 매개 변수가 몇 가지 작은 차이점이 있습니다. 매개 변수가 처음 것으로 간주 되어 할당 되지 않은 일부 값을 할당 하기 전에 멤버 본문에서 읽을 수 없습니다. 또한, 매개 변수 멤버 반환 되기 전에 일부 값이 할당 될 수 있습니다.  
  
 **X 방지** 를 사용 하 여 `out` 또는 `ref` 매개 변수입니다.  
  
 사용 하 여 `out` 또는 `ref` 매개 변수가 포인터를 어떻게 값 형식과 참조 형식이 다를 이해 하 고 반환 값이 여러 개인 메서드를 처리 해야 합니다. 또한 간의 차이 `out` 및 `ref` 매개 변수는 잘 알려져 있지 않습니다. 일반 사용자를 대상에 대 한 디자인 하는 프레임 워크 설계자와 작업에 능숙 할 사용자 기대 하지 않으면 `out` 또는 `ref` 매개 변수입니다.  
  
 **X 하지 않으려면** 참조 형식을 참조로 전달 합니다.  
  
 참조를 바꿔 사용할 수 있는 메서드 같은 규칙을 제한 된 몇 가지 예외가 있습니다.  
  
### 가변 개수의 매개 변수를 사용 하 여 멤버  
 가변 인수 수를 사용할 수 있는 멤버는 배열 매개 변수를 제공 하 여 표현 됩니다. 예를 들어 <xref:System.String> 다음 메서드를 제공 합니다.  
  
```  
public class String {  
    public static string Format(string format, object[] parameters);  
}  
```  
  
 사용자를 호출할 수는 <xref:System.String.Format%2A?displayProperty=fullName> 메서드를 다음과 같이 합니다.  
  
 `String.Format("File {0} not found in {1}",new object[]{filename,directory});`  
  
 C\# params 키워드는 배열 매개 변수에 추가 소위 params 배열 매개 변수를 매개 변수를 변경 하 고 임시 배열을 만드는 바로 가기를 제공 합니다.  
  
```  
public class String {  
    public static string Format(string format, params object[] parameters);  
}  
```  
  
 이 메서드를 호출 하 여 배열 요소를 인수 목록에 직접 전달 하 여 사용자 수 있습니다.  
  
 `String.Format("File {0} not found in {1}",filename,directory);`  
  
 참고 매개 변수 목록의 마지막 매개 변수에 params 키워드를 추가할 수 있습니다.  
  
 **✓ 고려** 는 적은 수의 요소 배열을 전달 하는 최종 사용자가 경우에 params 키워드 배열 매개 변수를 추가 합니다. 다양 한 요소 전달 될 공통 시나리오 될 것을 하는 경우 사용자는 이러한 요소 인라인 어쨌든 전달 하지 않을 이므로 params 키워드 아닙니다 필요 합니다.  
  
 **X 방지** 호출자는 거의 항상 입력에에서 이미 배열 하는 경우 매개 변수 배열을 사용 하 여 합니다.  
  
 예를 들어 개별 바이트를 전달 하 여 거의 없습니다 바이트 배열 매개 변수를 사용 하 여 멤버를 호출 합니다. 이러한 이유로.NET Framework의 바이트 배열 매개 변수는 params 키워드를 사용 하지 마십시오.  
  
 **X 하지 않으려면** 배열 매개 변수 배열 매개 변수를 사용 하는 멤버에 의해 수정 params 배열을 사용 합니다.  
  
 횟수가 많은 컴파일러에서는 호출 사이트에서 임시 배열에는 멤버에 대 한 인수를 설정, 배열 임시 개체 수 있고 따라서 배열에 수정이 손실 됩니다.  
  
 **✓ 고려** 좀 더 복잡 한 오버 로드를 사용 하지 못했습니다 하는 경우에 간단한 오버 로드에서 params 키워드를 사용 합니다.  
  
 하는 경우 사용자가 중요 하 게 생각 모든 오버 로드에는 되지 않은 경우에 한 오버 로드에 대 한 매개 변수 배열 포함 자문해 보십시오.  
  
 **✓ 수행** params 키워드를 사용할 수 있도록 적절 한 매개 변수를 하려고 합니다.  
  
 **✓ 고려** 적은 수의 성능이 매우 중요 한 Api에 대 한 인수를 사용 하 여 호출에 대 한 특별 한 오버 로드와 코드 경로 제공 합니다.  
  
 이렇게 하면는 적은 수의 인수는 API를 호출할 때 배열 개체를 만들지 않도록 하 고 있습니다. 배열 매개 변수는 단 수 형태의 서 숫자 접미사가 추가 매개 변수 이름을 형성 합니다.  
  
 만 이렇게 해야 경우 특별 한 경우 전체 코드 경로 배열을 만듭니다 뿐만 아니라 한 보다 일반적인 메서드를 호출 합니다.  
  
 **✓ 수행** 알고 그 null params 배열 인수로 전달 될 수 있어야 합니다.  
  
 배열의 처리 하기 전에 null 인지 유효성을 검사 해야 합니다.  
  
 **X 하지 마세요** 사용는 `varargs` 메서드를 가변 매개 변수 라고도 합니다.  
  
 C \+ \+와 같은 일부 CLR 언어 라는 변수 매개 변수 목록을 전달 하기 위한 대체 하는 규칙을 지원 `varargs` 메서드. CLS 규격 아니기 때문에 프레임 워크에서 규칙 해야 사용할 수 없습니다.  
  
### 포인터 매개 변수  
 일반적으로 포인터 되지 잘 설계 된 관리 코드 프레임 워크는 공개 노출 영역에 나타납니다. 대부분의 경우 포인터를 캡슐화 합니다. 그러나 포인터는 상호 운용성을 위해 필요한 경우에 따라 및 이러한 경우 포인터를 사용 하는 것이 적절 합니다.  
  
 **✓ 수행** 포인터를 CLS 규격 없기 때문에 포인터 인수를 사용 하는 멤버에 대 한 대체 방법을 제공 합니다.  
  
 **X 방지** 과도 한 인수 포인터 인수의 검사를 수행 합니다.  
  
 **✓ 수행** 포인터로 멤버를 디자인 하는 경우 일반 포인터와 관련 된 규칙을 따릅니다.  
  
 예를 들어, 간단한 포인터 산술을 동일한 결과 얻기 위해 사용할 수 있으므로 시작 인덱스를 전달할 필요가 없습니다 있습니다.
