 <span data-ttu-id="7354c-106">확장성을 방해 하는 강력한 메커니즘을 봉인 됩니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-106">A powerful mechanism that prevents extensibility is sealing.</span></span> <span data-ttu-id="7354c-107">클래스 또는 개별 멤버 봉인할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-107">You can seal either the class or individual members.</span></span> <span data-ttu-id="7354c-108">클래스 사용자를 클래스에서 상속할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-108">Sealing a class prevents users from inheriting from the class.</span></span> <span data-ttu-id="7354c-109">멤버를 봉인 사용자를에서 특정 멤버를 재정의할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-109">Sealing a member prevents users from overriding a particular member.</span></span>  
  
 <span data-ttu-id="7354c-110">**X 하지 않으면** 이유가 할 필요 없이 클래스를 봉인 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-110">**X DO NOT** seal classes without having a good reason to do so.</span></span>  
  
 <span data-ttu-id="7354c-111">확장성 시나리오를 생각할 수 때문에 클래스를 봉인 아닌 경우 하는 이유</span><span class="sxs-lookup"><span data-stu-id="7354c-111">Sealing a class because you cannot think of an extensibility scenario is not a good reason.</span></span> <span data-ttu-id="7354c-112">프레임 워크 사용자 편의 멤버를 추가 하는 등 다양 한 nonobvious 이유로 클래스에서 상속 하려고 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-112">Framework users like to inherit from classes for various nonobvious reasons, like adding convenience members.</span></span> <span data-ttu-id="7354c-113">참조 [봉인 되지 않은 클래스](../../../docs/standard/design-guidelines/unsealed-classes.md) nonobvious 대 한 예에 대 한 사용자 확인 하려는 형식에서 상속 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-113">See [Unsealed Classes](../../../docs/standard/design-guidelines/unsealed-classes.md) for examples of nonobvious reasons users want to inherit from a type.</span></span>  
  
 <span data-ttu-id="7354c-114">클래스에 대 한 좋은 이유는 다음과 같습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-114">Good reasons for sealing a class include the following:</span></span>  
  
-   <span data-ttu-id="7354c-115">클래스는 정적 클래스입니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-115">The class is a static class.</span></span> <span data-ttu-id="7354c-116">참조 [정적 클래스 디자인](../../../docs/standard/design-guidelines/static-class.md)합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-116">See [Static Class Design](../../../docs/standard/design-guidelines/static-class.md).</span></span>  
  
-   <span data-ttu-id="7354c-117">클래스 상속 된 보호 된 멤버에 중요 한 보안 암호를 저장합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-117">The class stores security-sensitive secrets in inherited protected members.</span></span>  
  
-   <span data-ttu-id="7354c-118">많은 가상 멤버를 상속 하는 클래스 및 봉인의 개별적으로 비용에는 봉인 되지 않은 클래스를 끝낼 이점 보다 높을 것입니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-118">The class inherits many virtual members and the cost of sealing them individually would outweigh the benefits of leaving the class unsealed.</span></span>  
  
-   <span data-ttu-id="7354c-119">클래스에는 매우 빠른 런타임 조회를 필요로 하는 특성입니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-119">The class is an attribute that requires very fast runtime look-up.</span></span> <span data-ttu-id="7354c-120">봉인 된 봉인 되지 않은 것 보다 약간 더 높은 성능 수준을 특성이 있습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-120">Sealed attributes have slightly higher performance levels than unsealed ones.</span></span> <span data-ttu-id="7354c-121">참조 [특성](../../../docs/standard/design-guidelines/attributes.md)합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-121">See [Attributes](../../../docs/standard/design-guidelines/attributes.md).</span></span>  
  
 <span data-ttu-id="7354c-122">**X 하지 않으면** 보호 또는 가상 멤버를 sealed 형식으로 선언 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-122">**X DO NOT** declare protected or virtual members on sealed types.</span></span>  
  
 <span data-ttu-id="7354c-123">기본적으로 sealed 형식에서 상속할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-123">By definition, sealed types cannot be inherited from.</span></span> <span data-ttu-id="7354c-124">즉, sealed 형식에 대해 보호 된 멤버를 호출할 수 없습니다, 봉인 된 형식에 대 한 가상 메서드를 재정의할 수 없습니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-124">This means that protected members on sealed types cannot be called, and virtual methods on sealed types cannot be overridden.</span></span>  
  
 <span data-ttu-id="7354c-125">**✓ 고려** 재정의할 수 있는 멤버가 봉인 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-125">**✓ CONSIDER** sealing members that you override.</span></span>  
  
 <span data-ttu-id="7354c-126">가상 멤버 소개에서 발생할 수 있는 문제 (에 설명 된 [가상 멤버](../../../docs/standard/design-guidelines/virtual-members.md)) 있지만에 약간도 재정의를 적용 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-126">Problems that can result from introducing virtual members (discussed in [Virtual Members](../../../docs/standard/design-guidelines/virtual-members.md)) apply to overrides as well, although to a slightly lesser degree.</span></span> <span data-ttu-id="7354c-127">재정의가 봉인 하면 상속 계층 구조에서 해당 시점에서 시작 하는 이러한 문제 로부터 보호 합니다.</span><span class="sxs-lookup"><span data-stu-id="7354c-127">Sealing an override shields you from these problems starting from that point in the inheritance hierarchy.</span></span>
