 <span data-ttu-id="77652-109">성능을 향상 시키기 위해 Tester-doer 패턴 나 다음 두 섹션에서 설명 하는 Try 구문 분석 패턴을 사용 하는 것이 같습니다.</span><span class="sxs-lookup"><span data-stu-id="77652-109">To improve performance, it is possible to use either the Tester-Doer Pattern or the Try-Parse Pattern, described in the next two sections.</span></span>  
  
## <a name="tester-doer-pattern"></a><span data-ttu-id="77652-110">Tester-doer 패턴</span><span class="sxs-lookup"><span data-stu-id="77652-110">Tester-Doer Pattern</span></span>

```  
ICollection<int> numbers = ...   
numbers.Add(1);  
```  
  
 <span data-ttu-id="77652-113">메서드가 `Add` 컬렉션이 읽기 전용인 경우 throw 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-113">The method `Add` throws if the collection is read-only.</span></span> <span data-ttu-id="77652-114">이 자주 실패 하 고 메서드 호출 필요한 시나리오에서 성능 문제를 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="77652-114">This can be a performance problem in scenarios where the method call is expected to fail often.</span></span> <span data-ttu-id="77652-115">문제를 완화 하는 방법 중 하나는 컬렉션 값을 추가 하기 전에 쓰기 가능 인지를 테스트 하는 것입니다.</span><span class="sxs-lookup"><span data-stu-id="77652-115">One of the ways to mitigate the problem is to test whether the collection is writable before trying to add a value.</span></span>  
  
```  
ICollection<int> numbers = ...   
...  
if(!numbers.IsReadOnly){  
    numbers.Add(1);  
}  
```  
  
 <span data-ttu-id="77652-116">속성은 예에서 사용 하는 조건을 테스트 하는 데 사용 하는 멤버 `IsReadOnly`은 테스터 라고 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-116">The member used to test a condition, which in our example is the property `IsReadOnly`, is referred to as the tester.</span></span> <span data-ttu-id="77652-117">잠재적으로 발생 작업을 수행 하는 데 사용 되는 멤버는 `Add` 예에서 메서드 doer 부분으로 참조 됩니다.</span><span class="sxs-lookup"><span data-stu-id="77652-117">The member used to perform a potentially throwing operation, the `Add` method in our example, is referred to as the doer.</span></span>  
  
 <span data-ttu-id="77652-118">**✓ 고려** Tester-doer 패턴 예외를 throw 할 수 있는 멤버에 대 한 공통 성능 문제를 방지 하는 시나리오와 관련 된 예외입니다.</span><span class="sxs-lookup"><span data-stu-id="77652-118">**✓ CONSIDER** the Tester-Doer Pattern for members that might throw exceptions in common scenarios to avoid performance problems related to exceptions.</span></span>  
  
## <a name="try-parse-pattern"></a><span data-ttu-id="77652-119">시도 구문 분석 패턴</span><span class="sxs-lookup"><span data-stu-id="77652-119">Try-Parse Pattern</span></span>  
 <span data-ttu-id="77652-120">성능이 매우 중요 한 Api에 대 한 이전 섹션에서 설명한 Tester-doer 패턴 보다 훨씬 빠른는 패턴을 사용 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-120">For extremely performance-sensitive APIs, an even faster pattern than the Tester-Doer Pattern described in the previous section should be used.</span></span> <span data-ttu-id="77652-121">멤버 이름에 멤버 의미 체계의 일부 경우 잘 정의 된 테스트를 조정 하기 위한 패턴을 호출 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-121">The pattern calls for adjusting the member name to make a well-defined test case a part of the member semantics.</span></span> <span data-ttu-id="77652-122">예를 들어 <xref:System.DateTime> 정의 <xref:System.DateTime.Parse%2A> 메서드는 실패 하는 문자열을 구문 분석 하는 경우 예외를 throw 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-122">For example, <xref:System.DateTime> defines a <xref:System.DateTime.Parse%2A> method that throws an exception if parsing of a string fails.</span></span> <span data-ttu-id="77652-123">또한 해당 정의 <xref:System.DateTime.TryParse%2A> , 구문 분석을 시도 하는 하지만 메서드 구문 분석 실패 하 고 사용 하 여 성공적으로 구문 분석의 결과 반환 하는 경우는 `out` 매개 변수입니다.</span><span class="sxs-lookup"><span data-stu-id="77652-123">It also defines a corresponding <xref:System.DateTime.TryParse%2A> method that attempts to parse, but returns false if parsing is unsuccessful and returns the result of a successful parsing using an `out` parameter.</span></span>  
  
```  
public struct DateTime {  
    public static DateTime Parse(string dateTime){   
        ...   
    }  
    public static bool TryParse(string dateTime, out DateTime result){  
        ...  
    }  
}  
```  
  
 <span data-ttu-id="77652-124">이 패턴을 사용할 경우에 엄격한 측면에서 시도 기능을 정의 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-124">When using this pattern, it is important to define the try functionality in strict terms.</span></span> <span data-ttu-id="77652-125">멤버 잘 정의 된 try 이외의 어떤 이유로 든 실패할 경우 멤버에 해당 하는 예외가 throw 여전히 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-125">If the member fails for any reason other than the well-defined try, the member must still throw a corresponding exception.</span></span>  
  
 <span data-ttu-id="77652-126">**✓ 고려** Try 구문 분석 패턴 예외를 throw 할 수 있는 멤버에 대 한 공통 성능 문제를 방지 하는 시나리오와 관련 된 예외입니다.</span><span class="sxs-lookup"><span data-stu-id="77652-126">**✓ CONSIDER** the Try-Parse Pattern for members that might throw exceptions in common scenarios to avoid performance problems related to exceptions.</span></span>  
  
 <span data-ttu-id="77652-127">**✓ 않습니다** 이 패턴을 구현 하는 방법에 대 한 접두사 "Try" 및 Boolean 반환 형식을 사용 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-127">**✓ DO** use the prefix "Try" and Boolean return type for methods implementing this pattern.</span></span>  
  
 <span data-ttu-id="77652-128">**✓ 않습니다** Try 구문 분석 패턴을 사용 하 여 각 멤버에 대 한 예외 throw 멤버를 제공 합니다.</span><span class="sxs-lookup"><span data-stu-id="77652-128">**✓ DO** provide an exception-throwing member for each member using the Try-Parse Pattern.</span></span>
