# <a name="operator-overloads"></a><span data-ttu-id="05009-102">연산자 오버로드</span><span class="sxs-lookup"><span data-stu-id="05009-102">Operator Overloads</span></span>
<span data-ttu-id="05009-103">연산자 오버 로드에는 것 처럼 기본 제공 언어 기본 형식 표시 프레임 워크 형식을 허용 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-103">Operator overloads allow framework types to appear as if they were built-in language primitives.</span></span>  
  
 <span data-ttu-id="05009-104">허용 되 고 일부 경우에 유용 하지만 연산자 오버 로드를 신중 하 게 사용 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-104">Although allowed and useful in some situations, operator overloads should be used cautiously.</span></span> <span data-ttu-id="05009-105">In 연산자는 오버 로드에 되었습니다 악용, 프레임 워크 디자이너는 간단한 방법이 되어야 하는 작업에 대 한 연산자를 사용 하기 시작 하는 경우와 같은 많은 경우가 있습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-105">There are many cases in which operator overloading has been abused, such as when framework designers started to use operators for operations that should be simple methods.</span></span> <span data-ttu-id="05009-106">다음 지침을 시기 및 연산자 오버 로드를 사용 하는 방법을 결정 하는 데 수 해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-106">The following guidelines should help you decide when and how to use operator overloading.</span></span>  
  
 <span data-ttu-id="05009-107">**하지 말고 X** 기본 (기본 제공) 형식 느낌을 해야 하는 형식을 제외 하 고 연산자 오버 로드를 정의 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-107">**X AVOID** defining operator overloads, except in types that should feel like primitive (built-in) types.</span></span>  
  
 <span data-ttu-id="05009-108">**✓ 고려** 기본 형식 처럼 있어야 하는 형식에서 연산자 오버 로드를 정의 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-108">**✓ CONSIDER** defining operator overloads in a type that should feel like a primitive type.</span></span>  
  
 <span data-ttu-id="05009-109">예를 들어 <xref:System.String?displayProperty=nameWithType> 가 `operator==` 및 `operator!=` 정의 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-109">For example, <xref:System.String?displayProperty=nameWithType> has `operator==` and `operator!=` defined.</span></span>  
  
 <span data-ttu-id="05009-110">**✓ 않습니다** 번호를 나타내는 구조체에서 연산자 오버 로드를 정의 (예: <xref:System.Decimal?displayProperty=nameWithType>).</span><span class="sxs-lookup"><span data-stu-id="05009-110">**✓ DO** define operator overloads in structs that represent numbers (such as <xref:System.Decimal?displayProperty=nameWithType>).</span></span>  
  
 <span data-ttu-id="05009-111">**X 하지 않으면** 연산자 오버 로드를 정의 하는 경우 간단한 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-111">**X DO NOT** be cute when defining operator overloads.</span></span>  
  
 <span data-ttu-id="05009-112">연산자 오버 로드 중인 즉각적으로 드러나지 될 작업의 결과 경우에 유용 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-112">Operator overloading is useful in cases in which it is immediately obvious what the result of the operation will be.</span></span> <span data-ttu-id="05009-113">1을 뺍니다 할 수는 의미가 예를 들어 <xref:System.DateTime> 에서 다른 `DateTime` 를 살펴보고 한 <xref:System.TimeSpan>합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-113">For example, it makes sense to be able to subtract one <xref:System.DateTime> from another `DateTime` and get a <xref:System.TimeSpan>.</span></span> <span data-ttu-id="05009-114">그러나 논리 union 연산자 union의 두 가지 데이터베이스 쿼리를 사용 하 여 또는 시프트 연산자를 사용 하 여 스트림에 쓰기 위해 적합 하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-114">However, it is not appropriate to use the logical union operator to union two database queries, or to use the shift operator to write to a stream.</span></span>  
  
 <span data-ttu-id="05009-115">**X 하지 않으면** 연산자는 오버 로드를 정의 하는 형식은 피연산자 중 하나 이상을 아닌 경우 오버 로드를 제공 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-115">**X DO NOT** provide operator overloads unless at least one of the operands is of the type defining the overload.</span></span>  
  
 <span data-ttu-id="05009-116">**✓ 않습니다** 대칭 방식에서 연산자를 오버 로드 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-116">**✓ DO** overload operators in a symmetric fashion.</span></span>  
  
 <span data-ttu-id="05009-117">예를 들어 오버 로드는 `operator==`, 연산자도 오버 로드 해야는 `operator!=`합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-117">For example, if you overload the `operator==`, you should also overload the `operator!=`.</span></span> <span data-ttu-id="05009-118">마찬가지로, 재정의 한 경우는 `operator<`, 연산자도 오버 로드 해야는 `operator>`등입니다.</span><span class="sxs-lookup"><span data-stu-id="05009-118">Similarly, if you overload the `operator<`, you should also overload the `operator>`, and so on.</span></span>  
  
 <span data-ttu-id="05009-119">**✓ 고려** 각 오버 로드 된 연산자에 해당 하는 이름을 가진 메서드를 제공 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-119">**✓ CONSIDER** providing methods with friendly names that correspond to each overloaded operator.</span></span>  
  
 <span data-ttu-id="05009-120">일반적인 언어 연산자 오버 로드를 지원 하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-120">Many languages do not support operator overloading.</span></span> <span data-ttu-id="05009-121">연산자 오버 로드 하는 형식을 포함 동일한 기능을 제공 하는 적절 한 도메인별 이름을 보조 메서드는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-121">For this reason, it is recommended that types that overload operators include a secondary method with an appropriate domain-specific name that provides equivalent functionality.</span></span>

### <a name="overloading-operator-"></a><span data-ttu-id="05009-126">연산자를 오버 로드 = =</span><span class="sxs-lookup"><span data-stu-id="05009-126">Overloading Operator ==</span></span>

### <a name="conversion-operators"></a><span data-ttu-id="05009-129">변환 연산자</span><span class="sxs-lookup"><span data-stu-id="05009-129">Conversion Operators</span></span>  
 <span data-ttu-id="05009-130">변환 연산자는 다른 한 형식에서 변환할 수 있는 단항 연산자입니다.</span><span class="sxs-lookup"><span data-stu-id="05009-130">Conversion operators are unary operators that allow conversion from one type to another.</span></span> <span data-ttu-id="05009-131">연산자는 피연산자 또는 반환 형식에 정적 멤버로 정의 되어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-131">The operators must be defined as static members on either the operand or the return type.</span></span> <span data-ttu-id="05009-132">변환 연산자의 두 가지: 명시적 및 암시적입니다.</span><span class="sxs-lookup"><span data-stu-id="05009-132">There are two types of conversion operators: implicit and explicit.</span></span>  
  
 <span data-ttu-id="05009-133">**X 하지 않으면** 이러한 변환의 최종 사용자가 예상 되는 것은 명확 하 게 경우 변환 연산자를 제공 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-133">**X DO NOT** provide a conversion operator if such conversion is not clearly expected by the end users.</span></span>  
  
 <span data-ttu-id="05009-134">**X 하지 않으면** 형식의 도메인 외부에서 변환 연산자를 정의 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-134">**X DO NOT** define conversion operators outside of a type’s domain.</span></span>  
  
 <span data-ttu-id="05009-135">예를 들어 <xref:System.Int32>, <xref:System.Double>, 및 <xref:System.Decimal> 은 모든 숫자 형식 이지만 <xref:System.DateTime> 않습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-135">For example, <xref:System.Int32>, <xref:System.Double>, and <xref:System.Decimal> are all numeric types, whereas <xref:System.DateTime> is not.</span></span> <span data-ttu-id="05009-136">따라서 없어야 없는 변환 연산자를 한 `Double(long)` 에 `DateTime`합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-136">Therefore, there should be no conversion operator to convert a `Double(long)` to a `DateTime`.</span></span> <span data-ttu-id="05009-137">이러한 경우에는 생성자는 것이 좋습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-137">A constructor is preferred in such a case.</span></span>  
  
 <span data-ttu-id="05009-138">**X 하지 않으면** 변환이 손실 될 수 있는 경우에 암시적 변환 연산자를 제공 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-138">**X DO NOT** provide an implicit conversion operator if the conversion is potentially lossy.</span></span>  
  
 <span data-ttu-id="05009-139">예를 들어 있으면 안에서 암시적 변환이 `Double` 를 `Int32` 때문에 `Double` 더 보다 넓은 범위의 `Int32`합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-139">For example, there should not be an implicit conversion from `Double` to `Int32` because `Double` has a wider range than `Int32`.</span></span> <span data-ttu-id="05009-140">명시적 변환 연산자는 변환이 손실 될 수 있는 경우에 제공 될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-140">An explicit conversion operator can be provided even if the conversion is potentially lossy.</span></span>  
  
 <span data-ttu-id="05009-141">**X 하지 않으면** 암시적 캐스트에서 예외를 throw 합니다.</span><span class="sxs-lookup"><span data-stu-id="05009-141">**X DO NOT** throw exceptions from implicit casts.</span></span>  
  
 <span data-ttu-id="05009-142">변환을 수행 되 고 알 수 때문에 발생 하는 작업을 이해 하는 최종 사용자에 대 한 매우 어렵습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-142">It is very difficult for end users to understand what is happening, because they might not be aware that a conversion is taking place.</span></span>  
  
 <span data-ttu-id="05009-143">**✓ 않습니다** throw <xref:System.InvalidCastException?displayProperty=nameWithType> 하면 캐스트 연산자에 대 한 호출 손실 변환와 연산자의 계약 손실 변환을 허용 하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="05009-143">**✓ DO** throw <xref:System.InvalidCastException?displayProperty=nameWithType> if a call to a cast operator results in a lossy conversion and the contract of the operator does not allow lossy conversions.</span></span>
