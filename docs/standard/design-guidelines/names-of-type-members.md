# <a name="names-of-type-members"></a><span data-ttu-id="12139-102">형식 멤버의 이름</span><span class="sxs-lookup"><span data-stu-id="12139-102">Names of Type Members</span></span>
<span data-ttu-id="12139-103">형식은 메서드, 속성, 이벤트, 생성자 및 필드 등의 멤버로 이루어집니다.</span><span class="sxs-lookup"><span data-stu-id="12139-103">Types are made of members: methods, properties, events, constructors, and fields.</span></span> <span data-ttu-id="12139-104">다음 섹션에서는 형식 멤버 이름 지정에 대한 지침을 설명합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-104">The following sections describe guidelines for naming type members.</span></span>  
  
## <a name="names-of-methods"></a><span data-ttu-id="12139-105">메서드 이름</span><span class="sxs-lookup"><span data-stu-id="12139-105">Names of Methods</span></span>  
 <span data-ttu-id="12139-106">메서드가 작업을 수행하는 수단이기 때문에 디자인 지침에서는 메서드 이름이 동사 또는 동사구여야 합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-106">Because methods are the means of taking action, the design guidelines require that method names be verbs or verb phrases.</span></span> <span data-ttu-id="12139-107">또한 이 지침을 따르면 명사 또는 형용사구인 속성 및 형식 이름과 메서드 이름을 구분할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="12139-107">Following this guideline also serves to distinguish method names from property and type names, which are noun or adjective phrases.</span></span>  
  
 <span data-ttu-id="12139-108">**✓ DO**를 통해 동사 또는 동사구인 메서드 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-108">**✓ DO** give methods names that are verbs or verb phrases.</span></span>  
  
```  
public class String {  
    public int CompareTo(...);  
    public string[] Split(...);  
    public string Trim();  
}  
```  
  
## <a name="names-of-properties"></a><span data-ttu-id="12139-109">속성 이름</span><span class="sxs-lookup"><span data-stu-id="12139-109">Names of Properties</span></span>  
 <span data-ttu-id="12139-110">다른 멤버와 달리 속성은 명사구 또는 형용사 이름이 지정되어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-110">Unlike other members, properties should be given noun phrase or adjective names.</span></span> <span data-ttu-id="12139-111">속성이 데이터를 나타내기 때문에 속성 이름은 이를 반영해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-111">That is because a property refers to data, and the name of the property reflects that.</span></span> <span data-ttu-id="12139-112">PascalCasing은 속성 이름에 항상 사용됩니다.</span><span class="sxs-lookup"><span data-stu-id="12139-112">PascalCasing is always used for property names.</span></span>  
  
 <span data-ttu-id="12139-113">**✓ DO**는 명사, 명사구 또는 형용사를 사용하여 속성 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-113">**✓ DO** name properties using a noun, noun phrase, or adjective.</span></span>  
  
 <span data-ttu-id="12139-114">**X DO NOT**에는 다음 예제와 같이 "Get" 메서드의 이름과 일치하는 속성이 있습니다.</span><span class="sxs-lookup"><span data-stu-id="12139-114">**X DO NOT** have properties that match the name of "Get" methods as in the following example:</span></span>  
  
 `public string TextWriter { get {...} set {...} }`  
 `public string GetTextWriter(int value) { ... }`  
  
 <span data-ttu-id="12139-115">이 패턴은 일반적으로 속성이 실제로 메서드임을 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="12139-115">This pattern typically indicates that the property should really be a method.</span></span>  
  
 <span data-ttu-id="12139-116">**✓ DO**는 뒤에 "List" 또는 "Collection"이 붙은 단일 구를 사용하는 대신 컬렉션에서 항목을 설명하는 복수 구를 사용하여 컬렉션 속성의 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-116">**✓ DO** name collection properties with a plural phrase describing the items in the collection instead of using a singular phrase followed by "List" or "Collection."</span></span>  
  
 <span data-ttu-id="12139-117">**✓ DO**는 긍정 구문(`CantSeek` 대신 `CanSeek`)을 사용하여 부울 속성의 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-117">**✓ DO** name Boolean properties with an affirmative phrase (`CanSeek` instead of `CantSeek`).</span></span> <span data-ttu-id="12139-118">또한 필요에 따라 부울 속성에 "Is," "Can," 또는 "Has"를 전치사로 지정할 수 있지만 여기에 값을 추가해야 합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-118">Optionally, you can also prefix Boolean properties with "Is," "Can," or "Has," but only where it adds value.</span></span>  
  
 <span data-ttu-id="12139-119">**✓ CONSIDER**는 속성에 해당 형식으로 동일한 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-119">**✓ CONSIDER** giving a property the same name as its type.</span></span>  
  
 <span data-ttu-id="12139-120">예를 들어, 다음 속성은 현재 올바르게 `Color`라는 열거형 값을 설정하므로 속성 이름은 `Color`입니다.</span><span class="sxs-lookup"><span data-stu-id="12139-120">For example, the following property correctly gets and sets an enum value named `Color`, so the property is named `Color`:</span></span>  
  
```  
public enum Color {...}  
public class Control {  
    public Color Color { get {...} set {...} }  
}  
```  
  
## <a name="names-of-events"></a><span data-ttu-id="12139-121">이벤트 이름</span><span class="sxs-lookup"><span data-stu-id="12139-121">Names of Events</span></span>  
 <span data-ttu-id="12139-122">이벤트는 발생 중인 작업 또는 발생한 작업 중 하나를 가리킵니다.</span><span class="sxs-lookup"><span data-stu-id="12139-122">Events always refer to some action, either one that is happening or one that has occurred.</span></span> <span data-ttu-id="12139-123">따라서 메서드와 마찬가지로 이벤트는 동사를 사용하여 명명하고 동사 시제는 이벤트가 발생할 시간을 나타내는 데 사용됩니다.</span><span class="sxs-lookup"><span data-stu-id="12139-123">Therefore, as with methods, events are named with verbs, and verb tense is used to indicate the time when the event is raised.</span></span>  
  
 <span data-ttu-id="12139-124">**✓ DO**는 동사 또는 동사구를 사용하여 이벤트의 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-124">**✓ DO** name events with a verb or a verb phrase.</span></span>  
  
 <span data-ttu-id="12139-125">예를 들면 `Clicked`, `Painting`, `DroppedDown` 등입니다.</span><span class="sxs-lookup"><span data-stu-id="12139-125">Examples include `Clicked`, `Painting`, `DroppedDown`, and so on.</span></span>  
  
 <span data-ttu-id="12139-126">**✓ DO**를 통해 현재 및 과거 시제를 사용하여 이전 및 이후 개념을 포함한 이벤트 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-126">**✓ DO** give events names with a concept of before and after, using the present and past tenses.</span></span>  
  
 <span data-ttu-id="12139-127">예를 들어 창이 닫히기 전에 발생하는 닫기 이벤트는 `Closing`이라고 하고 창이 닫힌 후에 발생하는 닫기 이벤트는 `Closed`라고 합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-127">For example, a close event that is raised before a window is closed would be called `Closing`, and one that is raised after the window is closed would be called `Closed`.</span></span>  
  
 <span data-ttu-id="12139-128">**X DO NOT**은 "Before" 또는 "After" 접두사 또는 접미사를 사용하여 사전 및 사후 이벤트를 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="12139-128">**X DO NOT** use "Before" or "After" prefixes or postfixes to indicate pre- and post-events.</span></span> <span data-ttu-id="12139-129">위에서 설명한 대로 현재 및 과거 시제를 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-129">Use present and past tenses as just described.</span></span>  
  
 <span data-ttu-id="12139-130">**✓ DO**는 다음 예제와 같이 "EventHandler" 접미사를 사용하여 이벤트 처리기(이벤트 형식으로 사용된 대리자)의 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-130">**✓ DO** name event handlers (delegates used as types of events) with the "EventHandler" suffix, as shown in the following example:</span></span>  
  
 `public delegate void ClickedEventHandler(object sender, ClickedEventArgs e);`  
  
 <span data-ttu-id="12139-131">**✓ DO**는 이벤트 처리기에서 `sender` 및 `e`라는 두 개의 매개 변수를 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-131">**✓ DO** use two parameters named `sender` and `e` in event handlers.</span></span>  
  
 <span data-ttu-id="12139-132">보낸 사람 매개 변수는 이벤트를 발생시킨 개체를 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="12139-132">The sender parameter represents the object that raised the event.</span></span> <span data-ttu-id="12139-133">보낸 사람 매개 변수는 보다 구체적인 형식을 적용할 수 있더라도 일반적으로 `object` 형식입니다.</span><span class="sxs-lookup"><span data-stu-id="12139-133">The sender parameter is typically of type `object`, even if it is possible to employ a more specific type.</span></span>  
  
 <span data-ttu-id="12139-134">**✓ DO**는 "EventArgs" 접미사를 사용하여 이벤트 인수 클래스의 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-134">**✓ DO** name event argument classes with the "EventArgs" suffix.</span></span>  
  
## <a name="names-of-fields"></a><span data-ttu-id="12139-135">필드 이름</span><span class="sxs-lookup"><span data-stu-id="12139-135">Names of Fields</span></span>  
 <span data-ttu-id="12139-136">필드 명명 지침은 공용 및 보호된 고정 필드에 적용됩니다.</span><span class="sxs-lookup"><span data-stu-id="12139-136">The field-naming guidelines apply to static public and protected fields.</span></span> <span data-ttu-id="12139-137">내부 및 개인 필드는 지침에서 다루지 않고 공용 또는 보호된 인스턴스 필드는 [멤버 디자인 지침](../../../docs/standard/design-guidelines/member.md)에서 허용하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="12139-137">Internal and private fields are not covered by guidelines, and public or protected instance fields are not allowed by the [member design guidelines](../../../docs/standard/design-guidelines/member.md).</span></span>  
  
 <span data-ttu-id="12139-138">**✓ DO**는 필드 이름에 PascalCasing을 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-138">**✓ DO** use PascalCasing in field names.</span></span>  
  
 <span data-ttu-id="12139-139">**✓ DO**는 명사, 명사구 또는 형용사를 사용하여 필드 이름을 지정합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-139">**✓ DO** name fields using a noun, noun phrase, or adjective.</span></span>  
  
 <span data-ttu-id="12139-140">**X DO NOT**은 필드 이름에 접두사를 사용합니다.</span><span class="sxs-lookup"><span data-stu-id="12139-140">**X DO NOT** use a prefix for field names.</span></span>  
  
 <span data-ttu-id="12139-141">예를 들어 "g_" 또는 "s_"를 사용하여 고정 필드를 나타내지 마십시오.</span><span class="sxs-lookup"><span data-stu-id="12139-141">For example, do not use "g_" or "s_" to indicate static fields.</span></span>
