## <a name="what-was-the-app-doing"></a><span data-ttu-id="c61f6-108">앱이 수행 중이었던 작업</span><span class="sxs-lookup"><span data-stu-id="c61f6-108">What was the app doing?</span></span>  
 <span data-ttu-id="c61f6-109">스택 베이스에서 [Windows.UI.Xaml](https://msdn.microsoft.com/library/windows/apps/windows.ui.xaml.aspx) 네임스페이스의 프레임은 XAML 렌더링 엔진이 실행 중이었음을 나타냅니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-109">At the base of the stack, frames from the [Windows.UI.Xaml](https://msdn.microsoft.com/library/windows/apps/windows.ui.xaml.aspx) namespace indicate that the XAML rendering engine was running.</span></span>   <span data-ttu-id="c61f6-110">또한 <xref:System.Reflection.PropertyInfo.GetValue%2A?displayProperty=nameWithType> 메서드가 사용되었으므로 해당 메타데이터가 제거된 형식에 대해 속성 값의 리플렉션 기반 조회를 수행했음을 알 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-110">The use of the <xref:System.Reflection.PropertyInfo.GetValue%2A?displayProperty=nameWithType> method indicates a reflection-based lookup of a property’s value on the type whose metadata was removed.</span></span>  
  
 <span data-ttu-id="c61f6-111">메타데이터 지시문을 제공하는 첫 단계에서는 모든 속성에 액세스할 수 있도록 형식에 대해 `serialize` 메타데이터를 추가합니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-111">The first step in providing a metadata directive would be to add `serialize` metadata for the type so that its properties are all accessible:</span></span>

```  
  
## <a name="is-this-an-isolated-case"></a><span data-ttu-id="c61f6-112">사례의 격리 여부 확인</span><span class="sxs-lookup"><span data-stu-id="c61f6-112">Is this an isolated case?</span></span>  
 <span data-ttu-id="c61f6-113">이 시나리오에서는 데이터 바인딩에서 `ViewModel` 하나의 메타데이터가 불완전하면 나머지 항목의 메타데이터도 불완전할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-113">In this scenario, if data binding has incomplete metadata for one `ViewModel`, it may for others, too.</span></span>  <span data-ttu-id="c61f6-114">앱의 뷰 모델이 모두 `App.ViewModels` 네임스페이스에 포함되는 방식으로 코드를 작성한 경우에는 보다 일반적인 런타임 지시문을 사용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-114">If the code is structured in a way that the app’s view models are all in the `App.ViewModels` namespace, you could use a more general runtime directive:</span></span>

```  
  
## <a name="could-the-code-be-rewritten-to-not-use-reflection"></a><span data-ttu-id="c61f6-115">리플렉션을 사용하지 않도록 코드 다시 작성 가능 여부 확인</span><span class="sxs-lookup"><span data-stu-id="c61f6-115">Could the code be rewritten to not use reflection?</span></span>  
 <span data-ttu-id="c61f6-116">데이터 바인딩에서는 리플렉션을 많이 사용하므로 리플렉션을 사용하지 않도록 코드를 변경하기는 어렵습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-116">Because data binding is reflection-intensive, changing the code to avoid reflection isn’t feasible.</span></span>  
  
 <span data-ttu-id="c61f6-117">그러나 `ViewModel`을 XAML 페이지로 지정하여 도구 체인이 컴파일 타임에 속성 바인딩을 올바른 형식과 연결하고, 런타임 지시문을 사용하지 않고 메타데이터를 유지하도록 할 수는 있습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-117">However, there are ways to specify the `ViewModel` to the XAML page so that the tool chain can associate property bindings with the correct type at compile time and keep the metadata without using a runtime directive.</span></span>  <span data-ttu-id="c61f6-118">예를 들어 속성에 대해 [Windows.UI.Xaml.Data.BindableAttribute](https://msdn.microsoft.com/library/windows/apps/windows.ui.xaml.data.bindableattribute.aspx) 특성을 적용할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-118">For example, you could apply the [Windows.UI.Xaml.Data.BindableAttribute](https://msdn.microsoft.com/library/windows/apps/windows.ui.xaml.data.bindableattribute.aspx) attribute on properties.</span></span> <span data-ttu-id="c61f6-119">이렇게 하면 XAML 컴파일러가 필요한 조회 정보를 생성하고 Default.rd.xml 파일에서 런타임 지시문이 필요하도록 설정하지 않습니다.</span><span class="sxs-lookup"><span data-stu-id="c61f6-119">This causes the XAML compiler to generate the required lookup information and avoids requiring a runtime directive in the Default.rd.xml file.</span></span>
