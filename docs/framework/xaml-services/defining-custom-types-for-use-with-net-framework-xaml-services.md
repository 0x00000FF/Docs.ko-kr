 <span data-ttu-id="ab3b3-205">부분 신뢰에서 XAML을 로드 하는 응용 프로그램의 경우 <xref:System.Xaml.Permissions.XamlAccessLevel> API를 사용 하 여 액세스 수준 특성을 제어할 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-205">For an application that loads XAML under partial trust, you can control the access level characteristics by using the <xref:System.Xaml.Permissions.XamlAccessLevel> API.</span></span> <span data-ttu-id="ab3b3-206">또한 지연 메커니즘 (예: WPF 템플릿 시스템)은 액세스 수준 권한을 전파 하 고 최종 실행 시간 평가를 위해 유지할 수 있어야 합니다. 이는 <xref:System.Xaml.Permissions.XamlAccessLevel> 정보를 전달 하 여 내부적으로 처리 됩니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-206">Also, deferral mechanisms (such as the WPF template system) must be able to propagate any access level permissions and preserve them for the eventual run time evaluations; this is handled internally by passing the <xref:System.Xaml.Permissions.XamlAccessLevel> information.</span></span>  
  
### <a name="wpf-implementation"></a><span data-ttu-id="ab3b3-207">WPF 구현</span><span class="sxs-lookup"><span data-stu-id="ab3b3-207">WPF Implementation</span></span>  
 <span data-ttu-id="ab3b3-208">WPF XAML은 부분 신뢰 액세스 모델을 사용 합니다. 여기서 BAML이 부분 신뢰로 로드 되 면 BAML 소스인 어셈블리에 대 한 <xref:System.Xaml.Permissions.XamlAccessLevel.AssemblyAccessTo%2A>에 대 한 액세스가 제한 됩니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-208">WPF XAML uses a partial-trust access model where if BAML is loaded under partial trust, access is restricted to <xref:System.Xaml.Permissions.XamlAccessLevel.AssemblyAccessTo%2A> for the assembly that is the BAML source.</span></span> <span data-ttu-id="ab3b3-209">지연의 경우 WPF는 액세스 수준 정보를 전달 하기 위한 메커니즘으로 <xref:System.Xaml.IXamlObjectWriterFactory.GetParentSettings%2A?displayProperty=nameWithType>를 사용 합니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-209">For deferral, WPF uses <xref:System.Xaml.IXamlObjectWriterFactory.GetParentSettings%2A?displayProperty=nameWithType> as a mechanism for passing the access level information.</span></span>  
  
 <span data-ttu-id="ab3b3-210">WPF XAML 용어에서 *내부 형식은* 참조 XAML도 포함 하는 동일한 어셈블리에 의해 정의 되는 형식입니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-210">In WPF XAML terminology, an *internal type* is a type that is defined by the same assembly that also includes the referencing XAML.</span></span> <span data-ttu-id="ab3b3-211">이러한 형식은 `xmlns:local="clr-namespace:WPFApplication1"`와 같이 매핑의 어셈블리를 의도적으로 생략 하는 XAML 네임 스페이스를 통해 매핑될 수 있습니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-211">Such a type can be mapped through a XAML namespace that deliberately omits the assembly= portion of a mapping, for example, `xmlns:local="clr-namespace:WPFApplication1"`.</span></span>  <span data-ttu-id="ab3b3-212">BAML이 내부 형식을 참조 하 고 해당 형식에 `internal` 액세스 수준이 있는 경우 어셈블리에 대 한 `GeneratedInternalTypeHelper` 클래스가 생성 됩니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-212">If BAML references an internal type and that type has `internal` access level, this generates a `GeneratedInternalTypeHelper` class for the assembly.</span></span> <span data-ttu-id="ab3b3-213">`GeneratedInternalTypeHelper`하지 않으려면 액세스 수준을 `public` 사용 해야 합니다. 그렇지 않으면 관련 클래스의 수준을 별도의 어셈블리로 지정 하 고 해당 어셈블리를 종속 된 것으로 만들어야 합니다.</span><span class="sxs-lookup"><span data-stu-id="ab3b3-213">If you want to avoid `GeneratedInternalTypeHelper`, you either must use `public` access level, or must factor the relevant class into a separate assembly and make that assembly dependent.</span></span>
